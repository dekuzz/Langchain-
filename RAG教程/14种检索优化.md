# RAG 检索优化整理

这份笔记按“问题层 / 模型层 / 召回层 / 上下文层 / 系统层”重新整理 14 类检索优化。

重点观察四件事：

1. 它解决什么问题
    
2. 它用什么思路解决
    
3. 代码实际怎么跑
    
4. 为什么这样做
    
5. 怎么评估它是否真的有效
    

---

# 一、总框架

RAG 检索优化不是单点技巧，而是在完整链路的不同位置做增强：

```text
用户问题
  ↓
问题改写 / 问题扩展
  ↓
召回候选文档
  ↓
重排 / 过滤 / 压缩
  ↓
拼接上下文
  ↓
LLM 生成答案
```

所以 14 类优化可以分为：

```text
问题层优化
├─ Multi Query
├─ RAG Fusion
├─ HyDE
└─ Step-Back Prompting

模型层优化
├─ Embedding 微调
└─ Reranker 微调

召回层优化
├─ BM25 / Hybrid
├─ Parent Document Retriever
└─ RAPTOR

上下文层优化
├─ Reranker
├─ Contextual Compression
├─ n_chunks 调参
└─ Contextual Embeddings

系统层优化
└─ CRAG
```

# 二、问题层优化

## 1. Multi Query

### 解决的问题

用户原始问题表达太单一，可能和文档里的表达对不上。

比如用户问：

```text
美国经济怎么样？
```

但文档里写的是：

```text
美国经济增长动能放缓，通胀回落速度不及预期。
```

原问题太短，向量检索可能抓不到关键语义。

### 解决思路

让 LLM 把一个问题改写成多个不同角度的问题，再分别检索。

```text
原始问题
  ↓
LLM 生成 5 个改写问题
  ↓
每个问题分别检索
  ↓
合并去重
  ↓
得到更多候选 chunk
```

### 代码实际流程

```python
query_prompt = PromptTemplate(...)

retriever_from_llm = MultiQueryRetriever.from_llm(
    retriever=vector_db.as_retriever(),
    llm=llm,
    prompt=query_prompt
)

unique_docs = retriever_from_llm.invoke(question)
```

实际跑法是：

```text
question
  → prompt
  → llm 生成多个 query
  → vector retriever 分别检索
  → 合并 docs
```

### 为什么这样做

向量检索本质上依赖“问题 embedding”和“文档 embedding”的距离。问题换一种说法，embedding 位置也会变。多个 query 可以扩大搜索半径。

### 如何评估

重点看：

```text
原问题 top-k 是否命中正确 chunk
Multi Query 后 top-k 是否命中正确 chunk
```

如果 Multi Query 只是召回更多无关文本，命中率没提升，甚至答案变差，就说明问题改写质量不够好。

## 2. RAG Fusion

### 解决的问题

Multi Query 会召回多个结果，但只是合并还不够。多个 query 的结果需要重新排序。

### 解决思路

多个 query 分别检索后，用排名融合思想给文档重新打分。

```text
query1 → docA docB docC
query2 → docB docA docD
query3 → docA docE docB

如果 docA / docB 多次靠前出现
说明它们更可能是正确结果
```

### 代码实际流程

典型流程是：

```text
原问题
  ↓
LLM 生成多个 query
  ↓
每个 query 检索 top-k
  ↓
对结果做 RRF 排名融合
  ↓
返回融合后的 top-k
```

核心是类似：

```python
def reciprocal_rank_fusion(results, k=60):
    fused_scores = {}
    for docs in results:
        for rank, doc in enumerate(docs):
            fused_scores[doc] += 1 / (rank + k)
    return sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)
```

### 为什么这样做

一个 query 可能偏，但多个 query 如果都把某个 chunk 排在前面，这个 chunk 的可信度更高。

### 如何评估

不要只看召回数量，要看排序是否变好：

```text
top1 hit rate ：Top1 Hit Rate = Top1 命中的问题数 / 总问题数
MRR：所有问题的 1 / 第一个正确结果排名 的平均值
Recall@k：top-k 中找回的相关 chunk 数 / 所有相关 chunk 数
```

| 指标            | 关心什么            | 是否关心排序   | 适合评估   |
| ------------- | --------------- | -------- | ------ |
| Top1 Hit Rate | 第一条是不是对的        | 只关心第 1 名 | 第一名准确率 |
| MRR           | 第一个正确结果排多靠前     | 很关心排序    | 排序质量   |
| Recall@k      | top-k 里找回多少正确内容 | 不太关心排序   | 召回覆盖率  |

如果 RAG Fusion 让正确 chunk 从第 6 位提前到第 2 位，就是有效的；如果只是把更多 chunk 合并进来，但正确 chunk 排名没变，就是效果有限。

## 3. HyDE

### 解决的问题

用户问题和原文可能不像，但“答案”和原文更像。

比如：

```text
问题：全球经济有哪些风险？
```

文档里可能没有“有哪些风险”这句话，而是直接写：

```text
地缘政治冲突、通胀粘性、金融条件收紧将影响全球增长。
```

### 解决思路

先让 LLM 生成一个假想答案，再用这个假想答案去检索。

```text
用户问题
  ↓
LLM 生成 hypothetical document
  ↓
用 hypothetical document 做 embedding
  ↓
检索真实文档
```

### 代码实际流程

```text
question
  → prompt
  → llm 生成假设性回答
  → embedding 假设性回答
  → similarity_search
```

### 为什么这样做

因为文档 chunk 通常更像“陈述句”，而不是“疑问句”。HyDE 把问题转成接近文档风格的文本。

### 如何评估

重点比较：

```text
直接 question 检索的 Recall@k
HyDE 生成文本检索的 Recall@k
```

还要人工看 HyDE 生成内容是否跑偏。如果假想答案幻觉严重，会把检索方向带歪。

## 4. Step-Back Prompting

### 解决的问题

用户问得很具体，但答案需要先理解背后的上位概念。

比如：

```text
2023 年美元指数为什么变化？
```

它背后其实需要理解：

```text
影响美元指数的因素有哪些？
```

### 解决思路

先让 LLM 从具体问题中抽象出一个更通用的问题，再用原问题和抽象问题一起辅助检索。

```text
具体问题
  ↓
抽象成 step-back question
  ↓
检索背景逻辑
  ↓
结合原问题检索具体事实
```

### 代码实际流程

```text
question
  → step_back_prompt
  → llm 生成上位问题
  → 检索 step-back question
  → 检索 original question
  → 合并上下文
  → 生成答案
```

### 为什么这样做

原问题负责找“具体事实”，step-back 问题负责找“解释框架”。

### 如何评估

不要只看是否命中原始 chunk，还要看最终答案是否更完整。适合用：

```text
Recall@k
答案完整性评分
答案是否包含原因链条
人工评估
LLM-as-judge
```

# 三、模型层优化

## 5. Embedding 微调

### 解决的问题

通用 embedding 模型不一定懂特定领域。

金融、医学、法律、技术文档中，很多词的相关性是领域内才成立的。

### 解决思路

用领域 QA 数据、正负样本对、相似文本对训练 embedding 模型，让它更会判断领域语义相似。

```text
领域问题
  ↓
正确 chunk 作为正样本
错误 chunk 作为负样本
  ↓
微调 embedding model
  ↓
重新向量化文档
  ↓
重新建库
```

### 代码实际流程

```text
加载文档
  ↓
切 chunk
  ↓
加载原始 BGE embedding
  ↓
加载微调后的 BGE embedding
  ↓
分别建 Chroma 向量库
  ↓
同一批问题测试 hit rate
```

### 为什么这样做

第一阶段召回如果错了，后面的 reranker 和生成都救不回来。Embedding 微调是在提高最底层召回质量。

### 如何评估

核心指标：

```text
Recall@k
Hit Rate@k
MRR
```

尤其看：

```text
baseline embedding vs fine-tuned embedding
```

如果微调后 top1、top3、top5 命中率都上升，说明检索空间变好了。

## 6. Reranker 微调

### 解决的问题

召回结果里有很多“看起来相关，但不是最相关”的 chunk。

### 解决思路

训练 reranker 判断 query 和 candidate chunk 的相关性。

```text
query + candidate chunk
  ↓
reranker 输出 relevance score
  ↓
按分数重新排序
```

### 代码实际流程

```text
retriever 先召回 top-k 候选
  ↓
把 question 和每个 doc 成对输入 reranker
  ↓
得到 relevance_score
  ↓
排序
  ↓
返回 top-n
```

### 为什么这样做

Embedding 是粗匹配，reranker 是精匹配。reranker 能同时看 query 和 chunk 的完整文本，因此判断更细。

### 如何评估

重点看排序指标：

```text
top1 hit rate
MRR
NDCG@k
```

如果正确 chunk 原本在 top8，reranker 后排到 top1，这就是明显有效。

# 四、召回层优化

## 7. BM25 / Hybrid

### 解决的问题

纯向量检索对数字、专有名词、缩写、精确词不一定稳定。

比如：

```text
2023
CPI
美联储
10年期美债收益率
```

这些内容关键词匹配很重要。

### 解决思路

同时使用：

```text
向量检索：找语义相似
BM25：找关键词匹配
Hybrid：融合两者结果
```

### 代码实际流程

```text
加载 docs
  ↓
BM25Retriever.from_documents(docs)
  ↓
Chroma + BGE embedding 建向量检索器
  ↓
EnsembleRetriever([vector_retriever, bm25_retriever], weights=[...])
  ↓
混合检索
```

### 为什么这样做

语义相似和关键词匹配是互补的。

```text
向量检索：召回“意思相近”的内容
BM25：召回“词面匹配”的内容
```

### 如何评估

分别比较：

```text
Vector only
BM25 only
Hybrid
```

重点看 Hybrid 是否提升 Recall@k，以及是否减少“语义相似但事实不对”的误召回。

## 8. Parent Document Retriever

### 解决的问题

小 chunk 检索准，但上下文不完整；大 chunk 信息完整，但检索不准。

### 解决思路

小 chunk 用来检索，大 chunk 用来返回。

```text
parent document
  ↓
切成 child chunks
  ↓
child chunks 建向量索引
  ↓
检索命中 child
  ↓
返回 child 所属 parent
```

### 代码实际流程

```python
parent_splitter = RecursiveCharacterTextSplitter(chunk_size=2000)
child_splitter = RecursiveCharacterTextSplitter(chunk_size=500)

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=store,
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)

retriever.add_documents(docs)
retriever.invoke(question)
```

### 为什么这样做

检索时需要精准定位，回答时需要完整语境。Parent Document Retriever 把这两个目标拆开处理。

### 如何评估

除了 Recall@k，还要看：

```text
返回上下文是否完整
答案是否减少断章取义
上下文长度是否明显增加
生成答案是否更稳定
```

注意：它可能提升答案完整度，但不一定提升 chunk-level hit rate，因为返回的是 parent，不是原始 child。

## 9. RAPTOR

### 解决的问题

长文档中，答案可能分布在多个段落或章节。普通 chunk 检索只能找到局部信息。

### 解决思路

把文档构造成层级摘要树：

```text
底层：原始 chunk
中层：多个 chunk 的摘要
高层：多个摘要的摘要
```

检索时既可以搜原始 chunk，也可以搜摘要节点。

### 代码实际流程

```text
文档切 chunk
  ↓
embedding chunks
  ↓
UMAP 降维
  ↓
GaussianMixture 聚类
  ↓
LLM 总结每个 cluster
  ↓
摘要继续 embedding / 聚类 / 总结
  ↓
形成多层节点
  ↓
检索时一起检索原文和摘要
```

### 为什么这样做

普通 RAG 更擅长局部事实，RAPTOR 更适合宏观总结、跨章节问题。

### 如何评估

适合看：

```text
跨段落问题的答案完整性
是否覆盖多个信息点
LLM-as-judge 综合评分
人工评分
```

单纯 Recall@k 不一定充分，因为 RAPTOR 的价值常体现在总结型问题上。

# 五、上下文层优化

## 10. Reranker

### 解决的问题

召回阶段找到了候选 chunk，但排序不准。

### 解决思路

先粗召回，再精排序。

```text
retriever 召回 top-k
  ↓
reranker 对 query-doc 打分
  ↓
重排
  ↓
返回最相关 top-n
```

### 代码实际流程

```text
question
  ↓
vector retriever / hybrid retriever 召回 candidates
  ↓
reranker.compute_score 或 rerank API
  ↓
得到 relevance_score
  ↓
按分数排序
```

### 为什么这样做

向量检索只比较两个向量，信息被压缩了。reranker 直接比较完整文本，所以判断更细。

### 如何评估

最适合看排序指标：

```text
top1 hit rate
MRR
NDCG@k
```

如果最终只传给 LLM 3 个 chunk，那么 top3 的排序质量比 top10 的召回质量更重要。

## 11. Contextual Compression

### 解决的问题

检索出来的 chunk 太长，里面混有大量无关内容。

### 解决思路

根据问题，对检索结果进行过滤或压缩。

```text
retriever 返回 docs
  ↓
compressor 判断哪些内容有用
  ↓
删除无关句子 / 删除无关文档
  ↓
返回更干净的上下文
```

### 代码实际流程

常见 compressor：

```text
LLMChainExtractor：从文档中抽取相关句子
LLMChainFilter：判断整个文档是否保留
EmbeddingsFilter：用 embedding 相似度过滤
```

组合方式：

```python
compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=retriever
)

compressed_docs = compression_retriever.invoke(question)
```

### 为什么这样做

LLM 上下文窗口有限。无关内容越多，模型越容易注意力分散，甚至被错误信息干扰。

### 如何评估

不仅看检索命中率，还要看：

```text
压缩后 token 数减少多少
正确答案证据是否还保留
最终答案准确率是否提升
是否误删关键证据
```

压缩不是越狠越好，误删关键内容会直接伤害答案。

---

## 12. n_chunks 调参

### 解决的问题

返回多少个 chunk 最合适？

太少会漏信息，太多会引入噪声。

### 解决思路

固定其他条件，只改变 k，观察检索和生成效果。

```text
k=1
k=2
k=4
k=8
k=16
```

### 代码实际流程

```python
for k in [1, 2, 4, 8, 16]:
    docs = vector_db.similarity_search(question, k=k)
    evaluate(docs, answer)
```

完整生成评估则是：

```text
不同 k 检索 docs
  ↓
拼接 context
  ↓
LLM 生成答案
  ↓
对答案打分
```

### 为什么这样做

RAG 的上下文不是越多越好。多出来的 chunk 如果不相关，就是噪声。

### 如何评估

分两层：

```text
检索层：Recall@k / Hit Rate@k
生成层：答案准确率 / 忠实度 / 引用证据质量
```

如果 Recall@8 比 Recall@4 高，但最终答案反而下降，说明多出来的 chunk 带来了噪声。

## 13. Contextual Embeddings

### 解决的问题

chunk 被切出来后失去上下文，里面的代词、省略、承接关系不清楚。

比如：

```text
这一趋势仍将持续。
```

单独 embedding 时，这句话几乎没有完整语义。

### 解决思路

在 embedding 前，先让 LLM 根据全文给 chunk 生成一段上下文说明，再拼接到 chunk 前面。

```text
完整文档 doc
  ↓
切成很多 chunk
  ↓
对每个 chunk：
    把“全文 doc + 当前 chunk”发给 LLM
  ↓
LLM 生成一小段上下文说明
  ↓
把“上下文说明 + 原 chunk”拼起来
  ↓
再做 embedding
  ↓
存进向量库
```

### 代码实际流程

```text
DOCUMENT_CONTEXT_PROMPT 放全文
CHUNK_CONTEXT_PROMPT 放当前 chunk
  ↓
situate_context(doc, chunk)
  ↓
contextualized_chunk = context + chunk
  ↓
embedding / Chroma
```

### 为什么这样做

Embedding 模型只能看到当前输入。如果 chunk 本身语义不完整，就需要提前补全它的语境。

### 如何评估

比较：

```text
原始 chunk embedding 的 Recall@k
contextualized chunk embedding 的 Recall@k
```

同时人工检查生成的 context 是否准确。如果 context 本身错误，会污染 embedding。

# 六、系统层优化

## 14. CRAG

### 解决的问题

本地知识库可能没有答案，或者检索结果质量很差。

普通 RAG 会硬答，容易幻觉。

### 解决思路

先判断本地检索结果是否足够相关。如果不够，就改写问题并联网搜索。

```text
retrieve
  ↓
grade documents
  ↓
如果相关：generate
如果不相关：transform query → web search → generate
```

### 代码实际流程

LangGraph 流程一般是：

```text
retrieve node
  ↓
grade_documents node
  ↓
条件判断
  ├─ generate
  └─ transform_query → web_search → generate
```

### 为什么这样做

它给 RAG 加了一个“质量检查开关”。不是所有问题都应该强行用本地库回答。

### 如何评估

要看：

```text
检索质量判断是否准确
该搜索时是否搜索
不该搜索时是否误触发
最终答案是否更准确
是否减少无依据回答
```

适合用混合评估集：

```text
一部分问题答案在本地库里
一部分问题答案不在本地库里
```

# 七、检索优化结果应该怎么评估

## 1. 先分清你评估的是“检索”还是“生成”

检索优化不能只看最终答案。最好分两层评估：

```text
检索层评估：正确证据有没有被找回来
生成层评估：LLM 是否基于证据答对
```

因为有时：

```text
检索对了，但生成错了
检索错了，但模型靠常识答对了
```

这两种情况都不能说明检索器真的好。

---

## 2. 检索层核心指标

### Hit Rate@k

只要 top-k 里出现正确 chunk，就算命中。

```text
Hit Rate@k = 命中问题数 / 总问题数
```

适合判断：

```text
正确证据有没有被召回
```

### Recall@k

如果一个问题有多个相关 chunk，Recall@k 看找回了多少。

```text
Recall@k = 找回的相关 chunk 数 / 所有相关 chunk 数
```

适合多证据问题。

### MRR
多个查询结果的平均倒数排名。

看第一个正确 chunk 排在多靠前。

```text
MRR = 1 / 第一个正确结果的排名
```

如果正确 chunk 排第 1，得分 1；排第 5，得分 0.2。

适合评估排序质量。

### NDCG@k

如果不同文档相关性有强弱，NDCG 更合适。

```text
强相关 > 弱相关 > 无关
```

适合 reranker、RAG Fusion 这类排序优化。

---

## 3. 生成层核心指标

### Answer Accuracy

答案是否答对。

### Faithfulness

答案是否忠实于检索到的上下文。

### Context Precision

传给 LLM 的上下文里，有多少是真正有用的。
$$\text{Context Precision} = \frac{\sum_{k=1}^{n} ((前 k 个结果中相关 chunk 的数量 / k )\times v_k)}{\text{检索结果中相关 Chunk 的总数}}$$
$$
vk:第 k 个 chunk 是否相关
$$
### Context Recall

回答问题需要的信息，有多少被上下文覆盖了。

### Token Cost

上下文 token 是否变多，成本是否增加。

---

## 4. 不同优化该重点看什么指标

|优化方法|重点指标|
|---|---|
|Embedding 微调|Recall@k / Hit Rate@k|
|Multi Query|Recall@k / 去重后有效召回数|
|RAG Fusion|MRR / NDCG@k / top1 hit rate|
|BM25 / Hybrid|Recall@k / 专有名词问题命中率|
|Reranker|MRR / NDCG@k / top1 hit rate|
|Reranker 微调|top1 hit rate / MRR|
|HyDE|Recall@k / 幻觉导致的误召回率|
|Step-Back|答案完整性 / 原因链条覆盖率|
|Parent Document Retriever|答案完整性 / 上下文覆盖率|
|Contextual Compression|token 减少率 / 证据保留率 / 答案准确率|
|n_chunks|Recall@k 与答案准确率的平衡点|
|RAPTOR|多跳问题准确率 / 摘要型问题完整性|
|Contextual Embeddings|Recall@k / 指代消解类问题命中率|
|CRAG|路由准确率 / 是否减少无依据回答|

# 八、最实用的实验设计

## 1. 准备评估集

每条数据最好有：

```json
{
  "question": "问题",
  "answer": "标准答案",
  "golden_chunk_id": "正确 chunk 的 id",
  "question_type": "事实型 / 总结型 / 多跳型 / 专有名词型"
}
```

如果没有 golden_chunk_id，至少要人工标注正确证据来源。

---

## 2. 固定变量，只改一个优化点

不要同时改太多东西。

错误做法：

```text
同时换 embedding、加 reranker、改 chunk_size、改 k
```

这样不知道是谁带来的提升。

正确做法：

```text
baseline
baseline + hybrid
baseline + reranker
baseline + multi query
baseline + contextual compression
```

---

## 3. 记录完整结果

建议每次实验记录：

```text
方法名
embedding 模型
chunk_size
chunk_overlap
top_k
是否 reranker
是否 compression
Recall@k
MRR
Answer Accuracy
平均 token 数
平均耗时
```

---

## 4. 最后用消融实验判断价值

比如完整系统是：

```text
Hybrid + Reranker + Contextual Compression
```

就分别去掉一个模块：

```text
去掉 Hybrid
去掉 Reranker
去掉 Compression
```

看哪个模块影响最大。

# 九、最终理解

检索优化不要理解成“14 个插件”。

更准确的理解是：

```text
问题层：让问题更好搜
模型层：让模型更懂领域
召回层：让候选文档更全
排序层：让正确文档更靠前
上下文层：让传给 LLM 的内容更干净
系统层：让 RAG 知道什么时候该自我纠错
```

真正做项目时，先跑 baseline，再看短板：

```text
召回不到 → 优化 embedding / multi query / hybrid
召回到了但排不前 → reranker / RAG Fusion
上下文太碎 → parent retriever
上下文太脏 → compression
文档太长 → RAPTOR
chunk 语义不完整 → contextual embeddings
知识库没有答案 → CRAG
```

评估的核心不是“用了多少高级方法”，而是：

```text
正确证据是否更容易被找回来
正确证据是否更靠前
无关上下文是否更少
最终答案是否更准确、更忠实、更省成本
```
