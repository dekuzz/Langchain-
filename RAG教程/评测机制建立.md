# 一、针对不同环节的指标
## 1. 检索层核心指标
检索层核心任务是“将包含正确答案的文档找回来，并排在前面”。
### **Hit Rate@k**: 
- 衡量底线召回能力，只要topk里包含正确的证据片段即为1，否则为0
### **MRR（平均倒数秩）**：
- 衡量正确证据的排序顺位，顺位越靠前，大模型生成的“迷失在中间”效应越低。

> [!NOTE] “迷失在中间”
> 当输入给大模型的上下文很长时，模型更容易关注开头和结尾的信息，而对放在中间的信息利用得更差。
### **NDCG@k(归一化累计折损增益)**:
- 适用于判断多篇不同相关度文档的综合排序质量。


## 2. 生成层核心指标
### **Faithfulness（忠诚度）**：
- 答案是否100%来自于检索到的上下文，有无产生外部幻觉
### **Answer Relevence(答案相关性)**：
- 答案是否直接回答了用户query
### **Context Relevence(上下文相关性)**：
- 检索回来的内容是否对回答问题有实际帮助

# 二、基于 TruLens 的自动化三元组评估
**TruLens** 是一个专门为 LLM 应用设计的评估框架。它最核心的理念是提出了 **“RAG 三元组 (RAG Triad)”** 评估：上下文相关性、忠实度、答案相关性。
## 核心原理：
内部使用LLM作为打分裁判，当内部代码调用`provider.groundedness_measure_with_cot_reasons`
`provider.relevance_with_cot_reasons`
`provider.context_relevance_with_cot_reasons`
时，框架向外部 Provider（如 OpenAI）发送的底层 Prompt 被强制植入了 CoT 指令，要求裁判模型在输出最终的数字分数（如 0-10 分）之前，必须先按步骤写下判断依据。客观数据证明，强制大模型先输出推理过程再输出结论，能有效降低裁判模型自身的打分幻觉。

``` python

# 1. 导入 TruLens 核心组件与反馈机制
from trulens_eval import Tru, TruChain, Feedback
from trulens_eval.feedback.provider import OpenAI
import numpy as np

# 初始化 TruLens 仪表盘
tru = Tru()
tru.reset_database()

# 2. 初始化 OpenAI 作为裁判提供者 (Provider)
provider = OpenAI()

# 3. 定义 RAG 三元组反馈函数 (Feedback Functions)

# 指标A：上下文相关性 (Context Relevance) - 评估检索质量
# 衡量 Query 与 检索到的 Context 之间的相关性
f_context_relevance = (
    Feedback(provider.context_relevance_with_cot_reasons, name="Context Relevance")
    .on_input() # 获取用户 Query
    .on(TruChain.select_context(rag_chain)) # 提取 RAG Chain 内部的 Context
    .aggregate(np.mean)# 给每个context的分数取平均
)

# 指标B：忠实度/事实一致性 (Groundedness/Faithfulness) - 评估幻觉
# 衡量最终 Answer 是否完全基于检索到的 Context
f_groundedness = (
    Feedback(provider.groundedness_measure_with_cot_reasons, name="Groundedness")
    .on(TruChain.select_context(rag_chain)) # 获取 Context
    .on_output() # 获取 Answer
)

# 指标C：答案相关性 (Answer Relevance) - 评估是否答非所问
# 衡量 Query 与最终 Answer 的相关性
f_answer_relevance = (
    Feedback(provider.relevance_with_cot_reasons, name="Answer Relevance")
    .on_input()  # 获取用户 Query
    .on_output() # 获取 Answer
)

# 4. 包装原始的 RAG Chain 进行监听评估
tru_recorder = TruChain(
    rag_chain,
    app_id='Baseline_RAG_v1', # 用于对照实验的版本号
    feedbacks=[f_context_relevance, f_groundedness, f_answer_relevance]
)

# 5. 执行提问并自动触发后台评估
answer_dict = {}  
  
for idx, row in tqdm(prediction_df.iterrows(), total=len(prediction_df)):  
    with tru_recorder as recording:  
        uuid = row['uuid']  
        question = row['question']  
        answer = rag_chain.invoke(question)  
        answer_dict[question] = {  
            'uuid': uuid,  
            'ref_answer': row['answer'],  
            'gen_answer': answer  
        }
        
        
prediction_df.loc[:, 'gen_answer'] = prediction_df['question'].apply(lambda q: answer_dict[q]['gen_answer'])

# 6. 启动本地 Dashboard 查看量化分数
tru.run_dashboard()
```

# 三、基于 GPT-as-a-Judge 的自定义评估
标准框架（如 TruLens）无法满足垂直领域特殊的评估需求（例如需要评估回答的专业语气、合规性或特定的推理逻辑）。此时，直接使用 **本地或者API的LLM作为裁判 (LLM-as-a-Judge)**，只针对答案正确性这一单一维度进行评估
## 核心原理：
评估能力完全来自于 Prompt 模板中赋予的“严谨客观的 RAG 系统评估专家”角色设定，以及明确的扣分基准定义。通过在上下文中提供完整的原始题干、证据和答卷，模型被强制在此闭环信息内进行模式匹配。

## 评估机制：
通过精心设计的 Prompt，向 LLM 传入 `Question`、`Context` 和 `Generated Answer`，并要求其按照指定的维度（如 1-5 分）进行打分，并强制输出 JSON 格式以便代码解析。

## 核心代码：
``` python
from langchain_openai import ChatOpenAI  
  
judge_llm = ChatOpenAI(  
    api_key=os.environ['API_KEY'],  
    base_url=os.environ['BASE_URL'],  
    # model_name='qwen2-72b-instruct'  
    # model_name='qwen-long' 0.68    model_name='gpt-4o-mini'  
)  
  
def evaluate(question, ref_answer, gen_answer):  
    """  
    对预测结果进行打分  
    :param question: 问题  
    :param ref_answer: 参考答案  
    :param gen_answer: 生成的答案  
    :return 打分模型原始返回结果  
    """    prompt = """  
你是一个经济学博士，现在我有一系列问题，有一个助手已经对这些问题进行了回答，你需要参照参考答案，评价这个助手的回答是否正确，仅回复“是”或“否”即可，不要带其他描述性内容或无关信息。  
问题：  
<question>  
{{question}}  
</question>  
  
参考答案：  
<ref_answer>  
{{ref_answer}}  
</ref_answer>  
  
助手回答：  
<gen_answer>  
{{gen_answer}}  
</gen_answer>  
请评价：  
    """.replace('{{question}}', question).replace('{{ref_answer}}', str(ref_answer)).replace('{{gen_answer}}', gen_answer).strip()  
    result = judge_llm.invoke(prompt).content  
    return result
    
score_dict = {}  
  
for idx, row in tqdm(prediction_df.iterrows(), total=len(prediction_df)):  
    question = row['question']  
    if question in score_dict:  
        continue  
    ref_answer = answer_dict[question]['ref_answer']  
    gen_answer = answer_dict[question]['gen_answer']  
    score_dict[question] = evaluate(question, ref_answer, gen_answer)

```

# 四、总结
一个严谨的 RAG 评估工程必须遵循以下实验设计原则：
### 4.1 构建黄金测试集 (Golden Dataset)

评估的第一步永远不是写代码，而是构建一个具有代表性的测试集。数据结构必须包含：

- `query`: 用户的典型提问。
    
- `ground_truth_context`: 正确的知识片段（用于评估检索召回率）。
    
- `ground_truth_answer`: 专家撰写的标准答案（用于比对生成的准确性）。
    

### 4.2 控制变量的消融实验

不要在一次实验中同时叠加多种优化（例如同时开启 HyDE 和 Reranker）。应该利用 `TruLens` 中的 `app_id` 进行版本对照：

1. **Baseline_RAG**: 仅基础分块 + 向量检索。
    
2. **RAG_v2_Hybrid**: Baseline + BM25 混合检索。
    
3. **RAG_v3_Reranker**: RAG_v2 + Cross-Encoder 重排。 通过对比不同 `app_id` 在仪表盘上的 `Context Relevance` 与 `Hit Rate` 的跳动幅度，客观量化每一层优化的实际 ROI（投资回报率）。

# 五、故障推演逻辑与评估总结 (Root Cause Analysis)

评估 RAG 以实现故障定责，核心是**强制解耦检索层与生成层**。必须依赖黄金验证集（包含 `query`、`golden_chunk_id`、`golden_answer`），按以下四步演绎逻辑进行精准故障推断。严禁跨级评估（Zero Trust 协议）。

### 5.1 第一步：验证基础召回物理能力 (白盒测试)

- **观测指标**：Hit Rate@k (命中率)
    
- **推论**：若 `Hit Rate@k` 极低，证明底层检索引擎或切块颗粒度彻底崩溃，目标信息未进入候选池。生成层评估此时毫无意义。
    
- **故障定责与优化**：
    
    1. **切块逻辑破裂**（如硬截断导致代词失联）：部署 `Parent Document Retriever` 或 `Contextual Embeddings` 还原语境。
        
    2. **句法与语义空间错位**（如短问长答、数值匹配失效）：部署 `HyDE` 抹平句法差异，或引入 `BM25 / Hybrid Search` 解决字面硬匹配。
        
    3. **模型领域认知盲区**：执行 Embedding 微调。
        

### 5.2 第二步：验证排序顺位与精排逻辑 (白盒测试)

- **观测指标**：MRR (平均倒数秩) / NDCG@k
    
- **前提**：`Hit Rate@k` 达标。
    
- **推论**：若 `MRR` 极低，证明粗排有效但缺乏细粒度对比能力，黄金证据沉底，必定触发大模型的“迷失在中间（Lost in the Middle）”效应。
    
- **故障定责与优化**：
    
    1. **双塔模型物理缺陷**：强制在向量初筛后串联 `Reranker`（交叉编码器）进行打分重排。
        
    2. **单路查询视角偏差**：部署 `Multi Query` 与 `RAG Fusion` 进行多路召回与倒数秩融合（RRF）。
        

### 5.3 第三步：验证上下文信噪比 (黑盒初筛)

- **观测指标**：Context Relevance (上下文相关性)
    
- **前提**：`Hit Rate` 与 `MRR` 均达标。
    
- **推论**：若该指标低，证明系统塞入了海量垃圾冗余信息，输入窗口被严重污染。
    
- **故障定责与优化**：
    
    1. **K值冗余/窗口过载**：执行 `n_chunks` 调参削减返回数量，或部署 `Contextual Compression` 物理剔除文档内部无关句子。
        

### 5.4 第四步：验证生成端忠实度与推理能力 (黑盒终测)

- **观测指标**：Faithfulness (忠实度) & Answer Relevance (答案相关性)
    
- **前提**：纯净证据已准确排在首位送达大模型。
    
- **推论**：若最终答案依然错误或产生幻觉，检索层无罪，故障完全归因于生成层自身缺陷或系统边界容错设计缺失。
    
- **故障定责与优化**：
    
    1. **知识库边界击穿**：部署 `CRAG` 路由，判定无关后直接触发外部 Web Search 补充事实。
        
    2. **宏观推理能力塌陷**：模型无法跨文档连线，退回检索层部署 `RAPTOR` 建立抽象摘要树，或部署 `Step-Back Prompting` 建立上位概念提取机制。
