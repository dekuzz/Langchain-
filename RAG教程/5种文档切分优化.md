# 为什么切块和检索相关？

当检索召回的 3 个 Chunk 长度极度不均匀（方差大，说明切分极其破碎，有些块只切出几十个字）时，上下文字段的信噪比会严重劣化，直接导致生成模型的回答准确率（score=0）断崖式下跌。

# 结构切块和递归切块
## 一、按文章结构切块的物理失控
- 核心论点：缺乏绝对长度约束的单一结构化切分，必然导致向量空间的物理扭曲和最终准确率的断崖式下跌。
###  **1. 物理数据的暴雷**
- **代码动作**：分别对 PDF 执行 `RecursiveCharacterTextSplitter` 和对 Markdown 执行 `MarkdownHeaderTextSplitter`，并调用 `describe()` 进行长度统计对比。
- **客观事实**：
	- 递归字符切分（PDF版）：均值 623，最大值 1306，标准差 258。
	- Markdown 标题切分（MD版）：均值 749，**最大值高达 2839，标准差 673**。
- **致命点**：
	- 代码中调用的 Embedding 模型是 `bge-large-zh-v1.5`，该模型底层的物理输入截断极限通常为 512 个 Token。长达 2839 个字符的文本块被送入该模型时，超过限制的后两千多字会被底层引擎**直接丢弃**。这导致生成的向量坐标仅仅代表该章节前 1/4 的语义，彻底摧毁了检索系统底层的数学依据。

### **3. 端到端准确率的崩塌**
- **代码动作**：引入 Qwen2-72B-instruct 作为裁判模型 (LLM-as-a-Judge)，对比参考答案 (`ref_answer`) 与生成答案 (`gen_answer`) 进行严格的 0/1 评分。
    
- **客观事实**：最终端到端平均得分 (`pred_df['score'].mean()`) **仅为 0.37 (37%)**。相较于常规基于长度限制的切片方式（上一份基线代码准确率为 0.72），发生了系统性的灾难退化。
    
- **致命点**： 这是“垃圾进，垃圾出 (Garbage In, Garbage Out)”在 RAG 中的绝对体现。检索环节提供的上下文要么缺失（未命中），要么充斥着无关的超长噪声，此时即便是 72B 级别的高参数裁判模型，也无法从错误的上下文中凭空捏造出正确答案。
``` python

import time  
  
judge_llm = ChatOpenAI(  
    api_key=os.environ['LLM_API_KEY'],  
    base_url=os.environ['LLM_BASE_URL'],  
    model_name='qwen2-72b-instruct',

)  
  
def evaluate(prediction_df):  
    """  
    对预测结果进行打分  
    :param prediction_df: 预测结果，需要包含问题，参考答案，生成的答案，列名分别为question, ref_answer, gen_answer

你是一个经济学博士，现在我有一系列问题，有一个助手已经对这些问题进行了回答，你需要参照参考答案，评价这个助手的回答是否正确，仅回复“是”或“否”即可，不要带其他描述性内容或无关信息。  
问题：  
<question>  
{{question}}  
</question>  
  
参考答案：  
<ref_answer>  
{{ref_answer}}  
</ref_answer>  
  
助手回答：  
<gen_answer>  
{{gen_answer}}  
</gen_answer>  
请评价：  
    """    results = []  
  
    for _, row in tqdm(prediction_df.iterrows(), total=len(prediction_df)):  
        question = row['question']  
        ref_answer = row['ref_answer']  
        gen_answer = row['gen_answer']  
  
        prompt = prompt_tmpl.replace('{{question}}', question).replace('{{ref_answer}}', str(ref_answer)).replace('{{gen_answer}}', gen_answer).strip()  
          
        retry_count = 3  
        result = ''  
        while retry_count > 0:  
            try:  
                result = judge_llm.invoke(prompt).content  
                break  
            except Exception as e:  
                retry_count -= 1  
                sleeping_seconds = 2 ** (4 - retry_count)  
                print(f"query={query}, error={e}, sleeping={sleeping_seconds}, remaining retry count={retry_count}")  
                  
                time.sleep(sleeping_seconds)  
          
        results.append(result)  
  
        time.sleep(1)  
    return results
```

## 二、级联切分

## 三、标题剥离与物理语义修复机制
### **1. 强制关闭标题剥离机制**
使用 `MarkdownHeaderTextSplitter` 时，强制追加参数 `strip_headers=False`
**原因**：
	*因为标题关键词对文本的语义方向起着决定性影响，所以必须设置 `strip_headers=False`。*

``` python
def split_md_docs(markdown_document):  
    headers_to_split_on = [  
        ("#", "Header 1"),  
        ("##", "Header 2"),  
        ("###", "Header 3"),  
    ]  
    markdown_splitter = MarkdownHeaderTextSplitter(headers_to_split_on, strip_headers=False)  
    md_header_splits = markdown_splitter.split_text(markdown_document)  
  
    return md_header_splits

```
### 2. **物理语义修复**
第二次切分的代码增加了一个强制遍历机制，提取 `small_chunks[1:]`（即第二块到最后一块）的 `metadata` 中的标题层级，将其重新拼接成 Markdown 格式（`### 标题`），并物理缝合（相加）到这些碎片的 `page_content` 头部。
**原因**：
	由于 `strip_headers=True`（默认），切分器会做出以下动作：

	1. 把 `### 2024年通胀预期` 塞进隐蔽的 `metadata` 字典中。
    
	2. 在真实的 `page_content` 中，**将这行标题彻底物理删除**。

``` python
from langchain.text_splitter import MarkdownTextSplitter  
  
new_md_splitted_docs = []  
splitter = MarkdownTextSplitter(  
    chunk_size=500,  
    chunk_overlap=50  
)  
for doc in md_splitted_docs:  
    if len(doc.page_content) > 700:  
        small_chunks = splitter.split_documents([doc])  
        # 把原始文档的标题回小片段的正文  
        for doc in small_chunks[1:]:  
            header_prefix = ''  
            for head_level in range(1, 4):  
                if f'Header {head_level}' in doc.metadata:  
                    header_prefix += '#' * head_level + ' ' + doc.metadata[f'Header {head_level}'] + '\n'  
            doc.page_content = header_prefix + doc.page_content  
          
        new_md_splitted_docs.extend(small_chunks)  
    else:  
        new_md_splitted_docs.append(doc)

```

# 语义切块+递归切块
- 语义切块通过余弦距离计算相邻句子的相似度，根据相似度突变位置进行切分
- 为了防止超长文本论述单一话题，从而导致相似度没有突变，对这种文本块进行二次切分，常常采用递归切分
``` python
from langchain_experimental.text_splitter import SemanticChunker
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceBgeEmbeddings

# 1. 语义切分阶段（理想主义）
embeddings = HuggingFaceBgeEmbeddings(model_name="BAAI/bge-large-zh-v1.5")
semantic_splitter = SemanticChunker(
    embeddings, 
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=95
)
semantic_chunks = semantic_splitter.create_documents([raw_text])

# 2. 机械暴力兜底阶段（现实主义防线）
# 不惜一切代价，确保进入 Embedding 引擎的文本长度绝对不越界
splitter = RecursiveCharacterTextSplitter(
    chunk_size=700,
    chunk_overlap=50,
    separators=['\n\n', '\n', '\s', '，']
)

def further_split(docs):
    ret_docs = []
    for doc in docs:
        if len(doc.page_content) >= 700:
            # 触发降级灾难：无视语义，挥舞剪刀强行斩断超限文本
            ret_docs.extend(splitter.split_documents([doc]))
        else:
            ret_docs.append(doc)
    return ret_docs

final_chunks = further_split(semantic_chunks)
```

# Jina Segment API 云端切块
## 定义：
	
## 优点：
- **用网络延迟换取本地算力与准确度**：
	- 相比于 `02_embedding_based_sementic_splitter.ipynb` 中需要本地显存/内存加载庞大 Embedding 模型来逐句计算余弦距离，API 调用的本地开销几乎为零。系统仅需极少的 CPU 资源维持网络连接。
- **高阶语义边界识别**：
	- 传统的 `RecursiveCharacterTextSplitter` 极易在小数点（3.14）、英文缩写（e.g., U.S.A., Dr.）等处发生误判断裂。Jina 的模型经过专门训练，能基于真实语境进行断句，避免了机械规则带来的语义割裂。
## 缺点：
- **信息暴露**：
	- 文本数据必须离开企业内网，全量暴露给第三方云服务商（Jina AI）。如果处理的是企业财务报表、用户隐私记录或核心技术文档，此举将直接触发严重的数据泄露与合规违约风险。**这是企业级私有化部署中的绝对红线。**
- **网络延迟影响大**：
	- 本地正则表达式切分百万字文本仅需几毫秒；本地 Embedding 切分需几分钟。而 HTTP 请求受到公网带宽、DNS 解析、TLS 握手及云端排队机制的制约。在处理超大规模文档集时，网络延迟将成为拖垮整个数据流水线的致命短板。
- **熔断风险**：
	- 对云端的切块逻辑没有任何控制权。一旦 API 规则更改、服务商宕机（502 Bad Gateway）或触发账户限流（Rate Limit / 429 Too Many Requests），整个建库流程将瞬间瘫痪，且无法在本地实施降级救援。

``` python
import os  
import pickle  
import requests  
from langchain.schema import Document  
  
def split_with_jina_api(text, max_len=700):  
    url = 'https://segment.jina.ai/'  
    headers = {  
        'Content-Type': 'application/json',  
        'Authorization': f"Bearer {os.getenv('JINA_SEGMENT_API_KEY')}"  
    }  
    data = {  
        "content": text,  
        "return_tokens": True,  
        "return_chunks": True,  
        "max_chunk_length": max_len  
    }  
    response = requests.post(url, headers=headers, json=data)  
    resp_json = response.json()  
    chunks = resp_json['chunks']  
    return [Document(page_content=chunk.strip()) for chunk in chunks if chunk.strip() != '']

```

# Meta-Chunking——基于大模型的逻辑感知切分

> [!NOTE] 注意

## 两种切分决策：
### **PPL Chunking(困惑度分块)：基于交叉熵损失的隐式预判**
- **判定逻辑**：直接利用模型自带的的预测下一个词能力来评估是否切块
使用时会面临的两种情况：
	预测平稳（不切分）：模型利用上一句话预测下一句话的准确概率高的话，此时交叉熵损失极小，困惑度（PPL :交叉熵指数）维持在低位
	预测突变（切分）：如果话题跳跃或者逻辑断裂，则预测准确概率低，此时交叉熵损失极高，困惑度维持高位
- **特征**：计算复杂度低，仅需一次常规的前向传播（Forward Pass）获取 Loss。
- **表现**：测试设备上单类文档耗时约 5 分钟，但下游检索与生成准确率（Accuracy）偏低（PDF 0.69，Markdown 0.64）。易受文本中无意义噪点（如表格结构、生僻词）干扰引发假性 PPL 激增。
- **适用场景**：适用于时效性要求极高、数据吞吐量庞大（TB级），且对长尾切分错误存在一定容忍度的粗粒度数据灌库任务。
- **核心代码实现**：

### **Margin Sampling Chunking（边缘采样分块）：基于显式推理的二元分类**
- **判定逻辑**：放弃模型的隐式概率分布，强制模型自己判断相邻两句话是否应该切分or合并，通过提取模型输出切分和合并两种选项的token概率，计算其差值。
- **特征**：抗噪能力强，语义完整性极高。通过深度逻辑理解，规避了语境截断引发的信息丢失或幻觉。
- **表现**：下游 RAG 问答准确率显著提升（PDF 0.78，Markdown 0.79）。**致命缺陷在于算力开销**：原设备（GTX 1080 Ti 11G）直接 OOM，改用 CPU 单文档耗时高达 5.5 小时。为每个句子组合构建 Prompt 并生成概率分布的成本呈指数级。
- **适用场景**：仅适用于**高价值、低频次**的核心知识资产解析（如金融招股书、研报、合规法律合同），针对此类场景，切分的准确率容错率必须为零。

``` python
    else:

        next_token_id_1 = output_ids[:, 1].unsqueeze(0)  
        next_token_prob_1 = token_probs[:, next_token_id_1].item()    
        prob_subtract=next_token_prob_1-next_token_prob_0  
    return prob_subtract
```

## 企业落地架构部署指导：
### **方案A：Margin Sampling 异步部署架构（针对高精度 / 零容错场景）**
**适用对象：** 深度审计报告、年度财务报表、内部核心合规制度。

**架构规约：** 绝对禁止将此逻辑挂载于同步处理链路。原实验数据证实该方法具有极端的计算时延与显存开销要求（GTX 1080 Ti 直接 OOM，CPU 耗时 5.5 小时）。若采用此方案以获取 0.79 的准确率，必须将其剥离为离线异步计算任务。必须通过消息队列（如 Kafka）进行批处理削峰，并部署独立的高端 GPU 节点（≥16GB 显存）结合专门的推理加速框架（如 `vLLM` 或 `TensorRT-LLM`）来专职处理 Token 概率提取。
### **方案 B：PPL Chunking 同步部署架构（针对高吞吐 / 成本敏感场景）**
**适用对象：** 日常运营数据、通用外部爬虫新闻、海量非核心文档库。

**架构规约：** 该方案可直接集成于常规数据中台的同步或准同步流水线中。利用其较低的计算复杂度（单文档处理耗时在分钟级），只需部署常规 CPU 或低端 GPU 集群即可满足大规模并发处理需求。企业在此架构中接受约 30% 左右的切块瑕疵容错率，以换取极高的数据流转速度与最低的算力成本。
### **共享前置约束：严格的文档预清洗机制**

无论采用上述方案 A 还是方案 B，进入切块算法层之前，必须强制执行严密的版面分析与数据清洗。必须通过版面分析引擎（如 MinerU）或正则清洗脚本（如原代码中强制剔除 `全球经济金融展望报告...` 等页眉）输出高纯度的连续文本流。若将未处理的噪点（离散页码、无意义表头）输入系统，将直接导致 PPL 计算出现无效激增，或严重污染 Margin Sampling 模型的分类置信度判定。
