# 通信协议的重要性

当我们每次需要访问新的外部服务时，我们都必须编写专门的tool，不仅工作量大，而且开发者的编写工具无法相互兼容。
**能力扩展的瓶颈**：智能体的能力被限制在预先定义的工具集内，无法动态发现和使用新的服务。
**协作的缺失**：当任务复杂到需要多个专业智能体协作时，我们只能通过手动编排来协调他们合作。
假设你要构建一个智能研究助手，它需要：
``` python
# 传统方式：手动集成每个服务
class GitHubTool(BaseTool):
    """需要手写GitHub API适配器"""
    def run(self, repo_url):
        # 大量的API调用代码...
        pass

class DatabaseTool(BaseTool):
    """需要手写数据库适配器"""
    def run(self, query):
        # 数据库连接和查询代码...
        pass

class WeatherTool(BaseTool):
    """需要手写天气API适配器"""
    def run(self, location):
        # 天气API调用代码...
        pass

# 每个新服务都需要重复这个过程
agent.add_tool(GitHubTool())
agent.add_tool(DatabaseTool())
agent.add_tool(WeatherTool())

```

这种方式会产生大量的代码重复（每个工具都需要处理HTTP请求、错误处理、认证），难以维护（API变更需要修改所有相关工具），无法复用（其他开发者的工具无法直接使用），扩展性差（添加新服务需要大量的代码工作）。

## MCP & A2A & ANP
### 1. MCP：智能体与工具的桥梁
由Anthropic提出
**核心设计理念**：
- 标准化智能体与外部工具/资源的通信方式。
![[智能体通信协议-1781145057099.webp]]
设计哲学是”上下文共享“，它不仅仅是一个 RPC（远程过程调用）协议，更重要的是它允许智能体和工具之间共享丰富的上下文信息。


### 2. A2A：智能体间的对话
由Google团队提出，
**核心设计理念**：
- 实现智能体之间点对点通信
A2A关注智能体之间的协作，让智能体像人类团队一样对话，协商和协作。

A2A的设计哲学是”对等通信“，在 A2A 网络中，每个智能体既是服务提供者，也是服务消费者。智能体可以主动发起请求，也可以响应其他智能体的请求。这种对等的设计避免了中心化协调器的瓶颈，让智能体网络更加灵活和可扩展。
![[智能体通信协议-1781145441322.webp]]

### 3. ANP：智能体网络的基础设施

由开源社区维护的概念性协议框架。
**核心设计理念**：
- 构建大型智能体网络基础设施。
如果说 **MCP** 解决的是"**如何访问工具**"，**A2A** 解决的是"**如何与其他智能体对话**"，那么 **ANP** 解决的是"**如何在大规模网络中发现和连接智能体**"。
**设计哲学**：
- ”去中心化服务发现“
ANP 提供了服务注册、发现和路由机制，让智能体能够动态地发现网络中的其他服务，而不需要预先配置所有的连接关系。
![[智能体通信协议-1781145675050.webp|692]]
### 4. 对比发现

| 维度   | MCP             | A2A        | ANP           |
| ---- | --------------- | ---------- | ------------- |
| 设计目标 | 智能体与工具/资源的标准化通信 | 智能体间的对点通信  | 大规模智能体网络的服务发现 |
| 通信模式 | 客户端-服务器（C/S）    | 对等网络（P2P）  | 对等网络（P2P）     |
| 核心理念 | 上下文共享           | 对等协作       | 去中心化发现        |
| 使用场景 | 访问外部工具和数据源      | 智能体协作和任务委托 | 大规模智能体生态系统    |
| 扩展性  | 添加MCP服务扩展       | 添加智能体节点扩展  | 支持动态扩展        |
| 实现状态 | FastMCP         | 官方SDK可用    | 概念性框架         |
# HelloAgents 通信协议架构设计
## 设计目标：
**让学习者能够以最简单的方式使用这些协议，同时保持足够的灵活性以应对复杂场景**。

## 通信协议架构：
### 三层设计：
- **智能体集成层**：
	- ToolSystem协议工具
- **工具封装层**：
	- 将所有的协议封装统一的Tool接口，MCPTool、A2ATool 和 ANPTool 都继承自 BaseTool，提供一致的`run()`方法。这种设计让智能体能够以相同的方式使用不同的协议。
- **协议实现层**：
	- MCP 基于 FastMCP 库实现，提供客户端和服务器功能；
	- A2A 基于 Google 官方的 a2a-sdk 实现；
	- ANP 是我们自研的轻量级实现，提供服务发现和网络管理功能，当然目前也有官方的[实现](https://github.com/agent-network-protocol/AgentConnect)，考虑到后期的迭代，因此这里只做概念的模拟。

# MCP协议实战
## 1. MCP架构
### Host:宿主层
- 用户直接交互的界面，它管理整个对话流程。
### Client：客户端层
- 与MCP建立连接，发送请求并接受响应。
### Services：服务层
- 实际的数据读取操作，并返回数据

![[智能体通信协议-1781147969529.webp]]
### 完整流程：
**一句话总结**：**Host 管理对话，Client 连接 Server，Server 暴露工具，LLM 根据工具描述选择工具，Client 通过 Server 执行工具，结果再返回给 LLM 生成最终回答。**
##### 1. 启动阶段：Host加载MCP配置
用户打开 Claude Desktop、Cursor、VS Code 这类 AI 应用。
这里的 AI 应用就是 **MCP Host**。
Host 会读取 MCP 配置，例如：
``` json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem"]
    }
  }
}

```
⬇
##### 2. 连接阶段：Host创建MCP Client
Host 不会直接和 MCP Server 交互，而是为每个 MCP Server 创建一个对应的 **MCP Client**。
``` Markdown
MCP Host
 ├── MCP Client 1 → 文件系统 MCP Server
 ├── MCP Client 2 → 数据库 MCP Server
 └── MCP Client 3 → GitHub MCP Server

```
##### 3. 工具发现阶段：Client获取工具列表
Client连接Server,获取server对应工具列表，server返回工具描述
##### 4. 上下文构建阶段：Client把工具信息交给LLM
Client 拿到工具列表后，会把这些工具以及针对工具的描述信息转成 LLM 能理解的格式。
**本质**：把 MCP Server 提供的工具，变成 LLM 推理时可见的上下文。
##### 5. 用户提问阶段：Host把问题给LLM
Host 会把用户问题、历史对话、工具列表等上下文一起发给 LLM。
##### 6. 模型推理阶段：LLM决定是否调用工具
工具选择不是 MCP Server 做的，也不是 MCP Client 做的，而是 LLM 根据工具描述做的。

##### 7. 工具执行阶段：Client 通过MCP Server 调用工具
LLM生成调用意图➡Client发送调用请求➡MCP Server
``` Markdown
LLM 负责决定
MCP Client 负责转发
MCP Server 负责执行
真实服务 / 文件 / API 负责提供数据

```

##### 8. 结果返回阶段：Server把结果交回Client
MCP Server 执行完后，会把结果返回给 Client。Client 再把这个工具结果送回给 LLM。
##### 9. 结果整合阶段：LLM生成最后回答


### 设计优势：
**关注点分离**：Host专注用户体验，Client专注协议通信，Server专注具体功能实现。
开发者只需要关注开发对应的MCP Server ,无需关心Host和Client的实现细节。

## 2.MCP核心能力
**MCP 协议提供了三大核心能力，构成完整的工具访问框架**

| 能力               | 说明           | 使用场景       | 示例                               |
| ---------------- | ------------ | ---------- | -------------------------------- |
| Tools（主动提供操作）    | 可执行能力，类似函数调用 | 执行操作、处理数据  | read_file,send_email,search_code |
| Resource（被动提供数据） | 可访问数据，类似文件系统 | 读取数据、订阅变化  | 文件内容、数据库记录、API响应                 |
| Prompt（指导性的提供模板） | 预定义提示模板      | 标准化任务、最佳实践 | 代码审查提示、文档生成提示                    |
## 3. Function Calling与MCP的差异

### 示例说明：
**Function Calling**
``` python
# 步骤1：为每个LLM提供商定义函数
# OpenAI格式
openai_tools = [
    {
        "type": "function",
        "function": {
            "name": "search_github",
            "description": "搜索GitHub仓库",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "搜索关键词"}
                },
                "required": ["query"]
            }
        }
    }
]

# Claude格式
claude_tools = [
    {
        "name": "search_github",
        "description": "搜索GitHub仓库",
        "input_schema": {  # 注意：不是parameters
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "搜索关键词"}
            },
            "required": ["query"]
        }
    }
]

# 步骤2：自己实现工具函数
def search_github(query):
    import requests
    response = requests.get(
        "https://api.github.com/search/repositories",
        params={"q": query}
    )
    return response.json()

# 步骤3：处理不同模型的响应格式
# OpenAI的响应
if response.choices[0].message.tool_calls:
    tool_call = response.choices[0].message.tool_calls[0]
    result = search_github(**json.loads(tool_call.function.arguments))

# Claude的响应
if response.content[0].type == "tool_use":
    tool_use = response.content[0]
    result = search_github(**tool_use.input)

```

**使用MCP**

### 定义的理解
#### Function Calling
让模型拥有告诉程序：*我需要什么工具，以及参数是什么的能力*，以结构化格式的方式。
#### MCP（Model Context Protocol）
让AI应用标准化连接外部工具、数据源和上下文的开放协议
**传统问题**：
1. **过往AI接入工具的方式**：
复制工具代码
复制工具描述
适配运行环境
写调用逻辑
处理返回结果

接入10个，重复10次这个步骤

2. **工具难以复用**：
- 语言不同
- 环境不同
- 接口格式不同

**MCP的核心思路：**
不要复制工具代码，而是让工具作为独立服务运行，AI 应用通过统一协议连接它。
它不是解决模型怎么思考，而是工具怎么被发现、描述、连接、调用、复用。

## 4. MCP传输方式
MCP本身不依赖特定传输方式，可在不同通信通道上运行。

| 传输方式           | 适用场景       | 优点          | 缺点             |
| -------------- | ---------- | ----------- | -------------- |
| Memory         | 单元测试、快速原型  | 最快、无网络开销    | 仅限同进程          |
| Stdio          | 本地开发、命令行工具 | 简单、无需网络配置   | 仅限本地、可能有兼容性问题  |
| HTTP           | 生产环境、远程服务  | 通用、防火墙友好    | 无流式支持、延迟较高     |
| SSE            | 实时通信、流式响应  | 支持服务器推送     | 单向通信、需要HTTP服务器 |
| StreamableHTTP | 流式HTTP通信   | 双向流式、HTTP兼容 | 需要特定服务器支持      |


### 4.1. Memory：同进程内调用
Memory不经过网络，也不启动子进程
Client和Server在同一个进程里
``` python
import asyncio
from fastmcp import FastMCP, Client

# 直接在内存里创建 MCP Server
mcp = FastMCP("MemoryDemo")


@mcp.tool
def add(a: int, b: int) -> int:
    """两个数字相加"""
    return a + b


async def main():
    # 直接把 mcp server 对象传给 Client
    client = Client(mcp)

    async with client:
        tools = await client.list_tools()
        print("工具列表:", [t.name for t in tools])

        result = await client.call_tool("add", {"a": 1, "b": 2})
        print("调用结果:", result)


asyncio.run(main())

```
### 4.2 Stdio:本地子进程通信，适合命令行工具 / Claude Desktop
#### server.py

``` python
from fastmcp import FastMCP
mcp = FastMCP("StdioDemo")
@mcp.tool
def add(a: int, b: int) -> int:    
"""两个数字相加"""    
return a + b
if __name__ == "__main__":    # 默认就是 stdio    
mcp.run()
```

#### client.py

``` python
import asyncio
from fastmcp import Client
async def main():    # 传入本地 Python 文件，Client 会用 stdio 启动它    
	client = Client("server.py") #创建 Client，准备使用 stdio 连接 server.py
	async with client: # 启动 server.py 子进程，并建立 stdin/stdout 管道     
		tools = await client.list_tools()        
		print("工具列表:", [t.name for t in tools])        
		result = await client.call_tool("add", {"a": 3, "b": 4})  
		"""
		→ Client 向 server.py 的 stdin 写请求  
		→ Server 从 stdout 返回响应  
		→ Client 解析后得到 result
		""" 
		print("调用结果:", result)
asyncio.run(main())
```

理解：

```
Client 启动 server.py 子进程Client 通过 stdin / stdout 和 Server 通信适合本地开发、命令行工具、Claude Desktop
```

FastMCP 文档里说明，STDIO transport 会通过子进程管道与 MCP server 通信，由 client 启动并管理 server 进程生命周期。

### 4.3: HTTP: 普通请求响应，适合理解传统远程 API

``` python
from pydantic import BaseModel
import uvicorn

app = FastAPI()
```


### 4.4. SSE：服务端单向流式推送，旧版 MCP 远程方案
**特点**：客户端发起连接，服务端持续往客户端推送数据。SSE 主要保留用于向后兼容，新项目更推荐 Streamable HTTP
sse_server.py
``` python
from fastmcp import FastMCP

mcp = FastMCP("SSEDemo")


@mcp.tool
def add(a: int, b: int) -> int:
    """两个数字相加"""
    return a + b


if __name__ == "__main__":
    # SSE 方式启动
    mcp.run(transport="sse", host="127.0.0.1", port=8000)

```
client.py
``` python
import asyncio
from fastmcp import Client
from fastmcp.client.transports import SSETransport


async def main():
    transport = SSETransport(
        url="http://127.0.0.1:8000/sse"
    )

    client = Client(transport)

    async with client:
        tools = await client.list_tools()
        print("工具列表:", [t.name for t in tools])

        result = await client.call_tool("add", {"a": 5, "b": 6})
        print("调用结果:", result)


asyncio.run(main())

```

### 4.5. StreamableHTTP：新版远程 MCP 推荐方案
它基于 HTTP，但支持更完整的双向通信、流式响应、多客户端连接，更适合生产部署。
streamable_http_server.py
``` python
from fastmcp import FastMCP

mcp = FastMCP("StreamableHTTPDemo")


@mcp.tool
def add(a: int, b: int) -> int:
    """两个数字相加"""
    return a + b


if __name__ == "__main__":
    # FastMCP 中 transport="http" 指 Streamable HTTP
    mcp.run(
        transport="http",
        host="127.0.0.1",
        port=8000
    )

```

client.py
``` python
import asyncio
from fastmcp import Client


async def main():
    client = Client("http://127.0.0.1:8000/mcp")

    async with client:
        tools = await client.list_tools()
        print("工具列表:", [t.name for t in tools])

        result = await client.call_tool("add", {"a": 7, "b": 8})
        print("调用结果:", result)


asyncio.run(main())
```
Server 是独立 HTTP 服务
Client 通过 URL 连接
支持远程访问
支持多个 Client
更适合生产部署

### 总结：
| 方式             | 代码上怎么体现                                                      | 适合场景                    |
| -------------- | ------------------------------------------------------------ | ----------------------- |
| Memory         | `Client(mcp)`                                                | 单元测试、快速原型               |
| Stdio          | `Client("server.py")` 或 `MCPClient(["python", "server.py"])` | 本地工具、Claude Desktop、命令行 |
| HTTP           | `requests.post("/add")`                                      | 传统 API，不是完整 MCP         |
| SSE            | `mcp.run(transport="sse")` + `SSETransport(...)`             | 旧版远程流式方案                |
| StreamableHTTP | `mcp.run(transport="http")` + `Client("http://.../mcp")`     | 新版远程 MCP 推荐方案           |
Memory：不出进程
Stdio：本地子进程
HTTP：普通远程请求响应
SSE：服务端单向流式推送
StreamableHTTP：新版 MCP 远程双向流式通信

（2）Memory-Transport 内存传输
适用场景：单元测试、快速原型开发
``` python
from hello_agents.tools import MCPTool

# 使用内置演示服务器（Memory传输）
mcp_tool = MCPTool()

# 列出可用工具
result = mcp_tool.run({"action": "list_tools"})
```

# A2A协议实战
A2A是一种智能体之间协作通信的协议
MCP解决了智能体和工具之间的交互，而A2A解决了智能体之间的协作问题。在一个多智能体协作的任务中，他们需要通信、委托任务、协商能力和同步状态
**传统中央协调器问题：**
- **单点故障**：协调器失效，整体系统瘫痪
- **性能瓶颈**：所有智能体都经过中心节点，限制并发
- **扩展困难**：增加或修改智能体需要修改中心逻辑
**A2A解决方案**：
采用P2P架构（网状拓补），允许智能体直接通信，从根本解决了问题。核心是**任务**和**工件**这两个抽象概念。
![[智能体通信协议-1781662733718.webp]]

**A2A标准生命周期**：
该机制使智能体可以进行任务协商、进度跟踪和异常处理
![[智能体通信协议-1781662819926.webp]]

A2A 请求生命周期是一个序列，详细说明了请求遵循的四个主要步骤：代理发现、身份验证、发送消息 API 和发送消息流 API

**（3）高级用法：Agent 间协商**

A2A 协议还支持 Agent 间的协商机制：

```python
from hello_agents.protocols import A2AServer, A2AClient
import threading
import time

# 创建两个需要协商的Agent
agent1 = A2AServer(
    name="agent1",
    description="Agent 1"
)

@agent1.skill("propose")
def handle_proposal(text: str) -> str:
    """处理协商提案"""
    import re
    
    # 解析提案
    match = re.search(r'propose\s+(.+)', text, re.IGNORECASE)
    proposal_str = match.group(1).strip() if match else text
    
    try:
        proposal = eval(proposal_str)
        task = proposal.get("task")
        deadline = proposal.get("deadline")
        
        # 评估提案
        if deadline >= 7:  # 至少需要7天
            result = {"accepted": True, "message": "接受提案"}
        else:
            result = {
                "accepted": False,
                "message": "时间太紧",
                "counter_proposal": {"deadline": 7}
            }
        return str(result)
    except:
        return str({"accepted": False, "message": "无效的提案格式"})

agent2 = A2AServer(
    name="agent2",
    description="Agent 2"
)

@agent2.skill("negotiate")
def negotiate_task(text: str) -> str:
    """发起协商"""
    import re
    
    # 解析任务和截止日期
    match = re.search(r'negotiate\s+task:(.+?)\s+deadline:(\d+)', text, re.IGNORECASE)
    if match:
        task = match.group(1).strip()
        deadline = int(match.group(2))
        
        # 向agent1发送提案
        proposal = {"task": task, "deadline": deadline}
        return str({"status": "negotiating", "proposal": proposal})
    else:
        return str({"status": "error", "message": "无效的协商请求"})

# 启动服务
threading.Thread(target=lambda: agent1.run(port=7000), daemon=True).start()
threading.Thread(target=lambda: agent2.run(port=7001), daemon=True).start()
```

# ANP 协议实战

[](https://github.com/datawhalechina/hello-agents/blob/main/docs/chapter10/%E7%AC%AC%E5%8D%81%E7%AB%A0%20%E6%99%BA%E8%83%BD%E4%BD%93%E9%80%9A%E4%BF%A1%E5%8D%8F%E8%AE%AE.md#104-anp-%E5%8D%8F%E8%AE%AE%E5%AE%9E%E6%88%98)

在 MCP 协议解决了工具调用、A2A 协议解决点对点智能体协作之后，ANP 协议则专注于解决大规模、开放网络环境下的智能体管理问题。

在 10.2 和 10.3 节中，我们学习了 MCP（工具访问）和 A2A（智能体协作）。现在，让我们学习 ANP（Agent Network Protocol）协议，它专注于构建**大规模、开放的智能体网络**。

### 10.4.1 协议目标

当一个网络中存在大量功能各异的智能体（例如，自然语言处理、图像识别、数据分析等）时，系统会面临一系列挑战：

- **服务发现**：当新任务到达时，如何快速找到能够处理该任务的智能体？
- **智能路由**：如果多个智能体都能处理同一任务，如何选择最合适的一个（如根据负载、成本等）并向其分派任务？
- **动态扩展**：如何让新加入网络的智能体被其他成员发现和调用？

ANP 的设计目标就是提供一套标准化的机制，来解决上述的服务发现、路由选择和网络扩展性问题。

为实现其设计目标，ANP 定义了以下几个核心概念，如表 10.8 所示：

表 10.8 ANP 核心概念

[![](https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/10-figures/10-table-8.png)](https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/10-figures/10-table-8.png)

[![](https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/10-figures/10-9.png)](https://raw.githubusercontent.com/datawhalechina/Hello-Agents/main/docs/images/10-figures/10-9.png)

图 10.9 ANP 整体流程

在这个流程图里，主要包括以下几个步骤：

**1. 服务的发现与匹配：**首先，智能体 A 通过一个公开的发现服务，基于语义或功能描述进行查询，以定位到符合其任务需求的智能体 B。该发现服务通过预先爬取各智能体对外暴露的标准端点（`.well-known/agent-descriptions`）来建立索引，从而实现服务需求方与提供方的动态匹配。

**2. 基于 DID 的身份验证：**在交互开始时，智能体 A 使用其私钥对包含自身 DID 的请求进行签名。智能体 B 收到后，通过解析该 DID 获取对应的公钥，并以此验证签名的真实性与请求的完整性，从而建立起双方的可信通信。

**3. 标准化的服务执行：**身份验证通过后，智能体 B 响应请求，双方依据预定义的标准接口和数据格式进行数据交换或服务调用（如预订、查询等）。标准化的交互流程是实现跨平台、跨系统互操作性的基础。

总而言之，该机制的核心是利用 DID 构建了一个去中心化的信任根基，并借助标准化的描述协议实现了服务的动态发现。这套方法使得智能体能够在无需中央协调的前提下，安全、高效地在互联网上形成协作网络。
