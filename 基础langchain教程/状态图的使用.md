==如果你不希望 Agent 拥有如此高的控制权，而是希望流程完全受你控制，那么你可以使用 **状态图**（StateGraph）来创建工作流。==

``` python
import os  
from dotenv import load_dotenv  
from langchain_openai import ChatOpenAI  
from langgraph.graph import StateGraph, MessagesState, START, END  
from langchain_core.messages import SystemMessage, HumanMessage  
from langchain_core.tools import tool  
  
# 关键节点：ToolNode  
from langgraph.prebuilt import ToolNode  
from langchain_core.runnables import RunnableConfig  
  
# 加载模型配置  
_ = load_dotenv()
# 加载模型  
llm = ChatOpenAI(  
    api_key=os.getenv("DASHSCOPE_API_KEY"),  
    base_url=os.getenv("DASHSCOPE_BASE_URL"),  
    model="qwen3-coder-plus",  
    temperature=0.7,  
)  
  
# 工具函数  
@tool  
def get_weather(city: str) -> str:  
    """Get weather for a given city."""  
    return f"It's always sunny in {city}!"  
  
# 创建工具节点  
tools = [get_weather]  
tool_node = ToolNode(tools)  
  
# 创建带工具的LLM助手节点  
def assistant(state: MessagesState, config: RunnableConfig):  
    system_prompt = 'You are a helpful assistant that can check weather.'  
    all_messages = [SystemMessage(system_prompt)] + state['messages']  
    model = llm.bind_tools(tools)  
    return {'messages': [model.invoke(all_messages)]}  
  
# 创建条件边  
def should_continue(state: MessagesState, config: RunnableConfig):  
    messages = state['messages']  
    last_message = messages[-1]  
    # 如果最后一条消息是LL决定使用工具调用，则继续  
    if last_message.tool_calls:  
        return 'continue'  
    # 否则，结束  
    return 'end'
    
# 创建图  
builder = StateGraph(MessagesState)  
  
# 添加节点  
builder.add_node('assistant', assistant)  
builder.add_node('tool', tool_node)  
  
# 添加边（从START到assistant）  
builder.add_edge(START, 'assistant')  
  
# 添加条件边（根据assistant的输出，决定是否调用工具）  
builder.add_conditional_edges(  
    # 条件边连接的节点    
'assistant',   
    # 条件边判断逻辑  
    should_continue,  
    # 条件边判断逻辑为True时，调用tool节点  
    {  
        'continue': 'tool',  
        'end': END,  
    },  
)  
  
# 添加边：调用工具节点后回到assistant  
builder.add_edge('tool', 'assistant')  
  
# 编译图  
my_graph = builder.compile(name='my-graph')  
my_graph
# 调用图  
response = my_graph.invoke({'messages': [HumanMessage(content='上海天气怎么样？')]})  
for message in response['messages']:  
    message.pretty_print()

```
# 1. StateGraph & MessagesState

**`StateGraph` 是“骨架/地图”，而 `MessagesState` 是“血液/载体”。**
## StateGraph :流程的设计图
- StateGraph是一个类，它负责定义工作流的结构
- **职责**：
	- **添加节点`add_node`**:
		- 注册一个节点及其处理逻辑
		- 【定义节点上有哪些工位（比如AI思考工位，工具执行工位）】
	- **定义路径`add_edge`**：
		- 定义状态流转关系，这里的**状态**特指`MessagesState` 里的消息状态
		- 【规定产品从A工位结束后，下一步去B工位还是C工位】
	- **编译流程 `compile`**：把这张设计图变成一个真正可以运行的程序。编译后，这个 `StateGraph` 才从“结构定义”变成“真正能 `invoke()` 执行的 agent 流程”
## MessagesState
- `MessagesState` 是一种特定的**数据结构（类型）**，它定义了节点之间传递的**数据长什么样**。
- **职责**：
    
    - **规定数据字段**：它告诉系统，我们的状态里有一个叫 `messages` 的列表。
        
    - **定义更新规则 (Reducer)**：它规定了当新消息产生时，是直接覆盖旧消息，还是把新消息追加到列表末尾（`MessagesState` 默认是追加）。
        
- **你可以把它理解为**：**搬运筐的规格**。它规定了在流水线上流转的那个“筐”里，只能装“消息列表”，而且新东西要放旧东西后面。

# 2.ToolNode 和`add_node`之间的关系
ToolNode是为了配合add_node而生的，通过创建一个ToolNode实例，再通过add_node把它部署到图纸上
如果只是用普通函数配合add_node处理工具，需要手写繁琐的json解析和状态更新代码，通过ToolNode这个早就把这些逻辑封装好的类，开箱即用。

## 使用方法
- 使用@tool定义工具
- 将所有工具 放到列表中
- 将工具绑定给大模型，让模型知道这些工具的存在
- 导入ToolNode,将工具列表传给它，生成一个执行对象，赋予系统执行工具的能力
- 通过StateGraph和add_node将大模型和工具都添加到生产线
- 设置条件路由，如果模型调用工具，就执行tool节点，如果普通聊天就结束
- 其中should_continue可以直接用官方的函数from langgraph.prebuilt import tools_condition替代
``` python
from langgraph.prebuilt import tools_condition
from langgraph.graph import END

# 添加条件边
builder.add_conditional_edges(
    source="assistant",          # 从大模型节点出来后进行判断
    path=tools_condition,        # 使用官方函数，它会自动检查最后一条消息有没有 tool_calls
    path_map={
        "tools": "tools",        # 如果有 tool_calls，流转到我们刚才定义的 "tools" 节点
        END: END                 # 如果没有，流转到结束
    }
)

# 工具执行完毕后，必须把结果传回给大模型继续思考
builder.add_edge("tools", "assistant")

# 编译图
graph = builder.compile()
```
